## 1. 项目概述

### 1.1 产品定位
一款以**语音输入**为核心交互方式的 AI 绘图工具。用户通过自然语言描述想要的画面，系统完成语音识别、意图理解、Prompt 优化和图像生成，并以语音 + 图像的方式反馈结果。

### 1.2 核心交互原则
- 绘图创作本身不依赖鼠标和键盘（生成、修改、撤销均由语音完成）。
- 录音环节允许使用“开始/结束录音”按钮，降低 Demo 录制风险。
- 系统具备**多轮对话能力**：信息不足时主动追问，生成后可基于语音继续修改。

### 1.3 核心能力
| 能力 | 说明 |
|------|------|
| 语音输入 | 录制用户语音并转为文本 |
| 意图理解 | 提取图像主体、风格、场景、色调、情绪、细节、构图等参数 |
| 完整性判断 | 信息不足时返回澄清问题，引导用户补充 |
| Prompt 扩写 | 将结构化参数转为高质量文生图 Prompt |
| 图像生成 | 调用文生图模型生成图片 |
| 迭代修改 | 基于当前图片上下文，语音调整局部元素后重新生成 |
| 文字反馈 | 通过聊天文字气泡展示执行结果或追问内容 |

### 3.4 ASR 详细方案：讯飞短语音识别 API

**选型理由**：
- 中文识别准确率高，对口音、口语化表达鲁棒性强。
- 国内网络访问稳定，延迟低，适合本地 Demo 和视频录制。
- 短语音识别 API 适合本项目“一句一段指令”的交互方式。

**接入方式**：
- 在讯飞开放平台（https://www.xfyun.cn）注册并创建应用，获取 `APPID`、`APIKey`、`APISecret`。
- 后端封装 `IflytekASRService`，统一对外暴露 `transcribe(audioBuffer: Buffer) -> string` 接口。
- 前端录音得到的音频格式（如 webm/opus）需在后端转换为 **PCM 16kHz 16bit 单声道**，再调用讯飞 API。

**音频格式转换**：
- 使用 `ffmpeg` 进行格式转换（Node.js 可通过 `fluent-ffmpeg` 调用）。
- 示例命令：
  ```bash
  ffmpeg -i input.webm -ar 16000 -ac 1 -f s16le output.pcm
  ```
- 也可先转 wav 再取 PCM 数据，视讯飞 SDK 要求而定。

**鉴权方式**：
- 讯飞短语音识别 WebAPI 使用 HMAC-SHA256 签名鉴权。
- 后端根据 `APPID`、`APIKey`、`APISecret` 生成请求 URL 和签名。

**返回结果**：
- API 返回 JSON，识别文本位于 `data.result` 或类似字段（以官方文档为准）。
- 后端提取文本后，进入 Stage 1 LLM。

**异常处理**：
- 识别为空或置信度过低时，返回 `error` 状态，前端提示用户重新录音。
- 网络失败时重试 1 次，仍失败则提示“识别服务暂时不可用”。

### 3.5 LLM 详细方案：DeepSeek

**选型理由**：
- DeepSeek-V3 / R1 中文理解和指令遵循能力强，适合 Stage 1 的参数提取和 Stage 2 的 Prompt 扩写。
- API 价格相对较低，适合多轮对话场景。
- 支持 JSON 模式 / 结构化输出，便于固定 Stage 1 的输出 Schema。

**Stage 1 与 Stage 2 是否共用同一模型**：
- 初期建议 Stage 1 和 Stage 2 都使用同一个 DeepSeek 模型，简化实现。
- 后续如果 Stage 2 对创意扩写要求更高，再考虑切换为更强的模型。

**最终选择**：
- **Stage 1 和 Stage 2 均使用 DeepSeek-V3**。
- 理由：通用能力强、速度快、价格低，足以满足参数提取和 Prompt 扩写需求。

**候选模型回顾**：
| 模型 | 特点 | 建议用途 |
|------|------|----------|
| DeepSeek-V3 | 通用能力强、速度快、成本低 | Stage 1 + Stage 2 首选（已选） |
| DeepSeek-R1 | 推理能力强，适合复杂语义分析 | Stage 1 复杂意图理解（备选） |
| DeepSeek-V2.5 | 更便宜，能力稍弱 | 成本敏感时备选 |

**接入方式**：
- 在 DeepSeek 开放平台（https://platform.deepseek.com）注册并获取 API Key。
- 模型名指定为 `deepseek-chat`（对应 DeepSeek-V3）。
- 后端封装 `DeepSeekService`，提供两个方法：
  - `extractParams(userInput, context)` → Stage 1 JSON
  - `expandPrompt(extracted)` → Stage 2 Prompt 字符串
- 调用 DeepSeek Chat Completions API，开启 JSON 模式（`response_format: { type: "json_object" }`）以保证 Stage 1 输出格式稳定。

**异常处理**：
- JSON 解析失败时，尝试让 LLM 重新生成 1 次。
- 仍然失败时返回 `error` 状态，提示用户“理解失败，请重新描述”。

### 3.6 图像生成详细方案：通义万相（Wanx）

**选型理由**：
- 阿里云生态稳定，API 文档成熟，适合 Demo 录制。
- 中文 Prompt 理解能力较好，也支持英文 Prompt。
- 不仅支持文生图，还支持基于原图的指令编辑，满足“基于原图修改”的需求。
- 提供多种尺寸和极速模式，方便控制生成耗时。

**首次生成：文生图 API**
- 模型：`wanx-v1` 或 `wanx2.1-t2i-turbo`（以官方最新为准）
- 调用方式：DashScope SDK 或 HTTP API
- 输入：Stage 2 生成的高质量 Prompt
- 输出：图片 URL

**后续修改：图像编辑 API**
- 模型：`wanx2.1-imageedit`
- 功能：`description_edit`（指令编辑，基于原图按指令修改）
- 输入：
  - `base_image_url`：当前图片 URL（即要修改的原图）
  - `prompt`：修改指令或完整画面描述
- 输出：编辑后的图片 URL
- 特点：在保留原图整体构图和元素的基础上，按指令进行局部或全局修改。

**关键参数**：
| 参数 | 说明 | 建议值 |
|------|------|--------|
| 文生图模型 | 首次生成使用 | `wanx-v1` 或官方最新版本 |
| 图像编辑模型 | 后续修改使用 | `wanx2.1-imageedit` |
| 尺寸 | 输出图片分辨率 | 1024×1024（默认），可选 1024×768 / 768×1024 |
| 模式 | 是否使用极速模式 | 极速模式优先，保证 Demo 体验 |
| Prompt | 生成提示词 | 由 Stage 2 LLM 输出 |

**返回结果**：
- API 返回图片 URL 或 base64 数据。
- 后端将图片 URL 返回前端展示。

**异常处理**：
- 生成失败时重试 1 次。
- 仍然失败则返回 `error` 状态，提示用户“图片生成失败，请重试”。
- 编辑失败时，可降级为“基于完整 Prompt 重新文生图”。

## 3.7 API Key 配置

前端左侧配置面板允许用户填写各类 API Key，保存在浏览器 `localStorage`。每次请求时通过 HTTP Header 发送给后端：

- `X-IFLYTEK-APP-ID`
- `X-IFLYTEK-API-KEY`
- `X-IFLYTEK-API-SECRET`
- `X-DEEPSEEK-API-KEY`
- `X-DEEPSEEK-MODEL`
- `X-DASHSCOPE-API-KEY`
- `X-DASHSCOPE-IMAGE-MODEL`

后端通过 Express 中间件读取 Header，并借助 Node.js `AsyncLocalStorage` 建立请求级配置上下文。所有服务层通过 `getApiConfig()` 读取配置，优先使用请求头中的值，其次回退到 `process.env`。Header 留空时完全沿用服务端 `.env` 默认值。

## 4. 核心流程

### 4.1 首次生成流程

1. 用户点击“开始录音”按钮并描述画面。
2. 用户点击“结束录音”，前端将音频上传至后端。
3. 后端 ASR 模块将语音转为文本。
4. 后端调用 **Stage 1 LLM**：
   - 提取结构化参数
   - 判断是否信息完整
   - 若缺失关键信息，返回 `need_clarification` 状态及追问问题
   - 若完整，返回 `complete` 状态及完整参数
5. 若需要澄清：
   - 前端在聊天中以文字气泡展示追问和建议示例
   - 用户再次录音补充，携带上下文重新进入 Stage 1
6. 信息完整后，进入 **Stage 2 LLM**，扩写为高质量文生图 Prompt。
7. 调用 **Stage 3 图像生成模型**，生成图片。
8. 前端展示图片，并以文字气泡展示结果说明。

### 4.2 迭代修改流程

1. 用户看到当前生成图片后，语音提出修改，例如：
   - “把猫改成黑色的。”
   - “背景换成星空。”
   - “更写实一点。”
   - “把猫向上放一点。”
2. 后端携带当前图片的参数上下文，再次进入 Stage 1。
3. LLM 根据修改请求更新对应字段，保持其他字段不变。
4. 进入 Stage 2，生成修改后的画面描述 Prompt。
5. 调用 **通义万相图像编辑 API**（`wanx2.1-imageedit`，`function: description_edit`），以当前图片为底图，按 Stage 2 的描述进行编辑。
6. 前端展示编辑后的新图片，并以文字气泡展示修改结果说明。

**例外**：如果用户明确说“整张重画”、“重新生成”、“不要这张了，重新画”等，则放弃图像编辑，回到文生图流程重新生成。

### 4.3 上下文管理

后端维护每个会话的上下文，至少包含：

```json
{
  "session_id": "uuid",
  "current_params": { /* 当前积累的结构化参数 */ },
  "current_image_url": "...",
  "conversation_history": [
    { "role": "user", "content": "...", "type": "voice" },
    { "role": "assistant", "content": "...", "type": "text" },
    { "role": "assistant", "content": "...", "type": "image" }
  ],
  "pending_clarification": false,
  "mode": "creating" | "modifying"
}
```

上下文作用：
- 支持多轮澄清：用户补充信息时，LLM 基于已有参数继续完善。
- 支持多轮修改：用户说“把猫改成黑色”时，LLM 知道当前画面是什么，只改对应部分。
- 支持“停止输入”语义：用户说“随便吧”、“就这样吧”、“直接画”时，用当前参数强制生成。

### 4.4 页面交互设计（微信对话式）

整体页面参考微信聊天界面：消息气泡上下排列，用户语音消息在右侧，AI 回复在左侧。

#### 4.4.1 消息类型

| 消息类型 | 发送方 | 展示形式 |
|----------|--------|----------|
| 语音输入 | 用户 | 语音气泡（显示时长 + 播放按钮 + 转写文字） |
| 文字提示 | AI | 文字气泡（追问、确认、结果说明） |
| 图片 | AI | 图片卡片（大图展示 + 重新生成 / 修改按钮） |
| 建议选项 | AI | 横向滚动的建议卡片，可点击或语音选择 |

#### 4.4.2 首次生成对话示例

**用户** [语音]：画一只猫。

**AI** [文字]：你想画一只什么样的猫？请补充一下风格和场景。
> 💡 建议：写实风格的橘猫，在窗台上晒太阳
> 💡 建议：卡通风格的黑猫，在太空里漂浮

**用户** [语音]：卡通风格的，在太空里。

**AI** [文字]：好的，一只卡通风格的太空猫，马上开始画。

**AI** [图片]：*[生成的卡通太空猫图片]*

#### 4.4.3 “停止输入”语义

当 LLM 还在追问时，用户可以说：
- “随便吧。”
- “就这样吧。”
- “直接画。”
- “不补充了。”

此时 Stage 1 LLM 应返回 `force_generate: true`，用当前已提取的参数直接生成图片。

Stage 1 输出需扩展字段：

```json
{
  "status": "complete",
  "force_generate": true,
  "extracted": { /* 当前参数，允许部分字段为 null */ },
  "response": "好的，我用当前信息直接为你生成图片。"
}
```

LLM 在缺失字段较多时，用默认值或合理推断补全（例如场景缺失则设为“简洁背景”），不再追问。

#### 4.4.4 修改图片对话示例

**用户** [语音]：把猫改成黑色的。

**AI** [文字]：好的，把猫改成黑色，其他保持不变。

**AI** [图片]：*[更新后的图片]*

#### 4.4.5 底部输入区

参考微信底部输入栏：
- 左侧：【语音/键盘切换按钮】（本项目只保留语音模式，可隐藏或作为装饰）
- 中间：【按住说话】大按钮，按住录音，松手发送
- 右侧：可选的【重新生成】按钮

按钮交互：
- 按住：开始录音，显示“松开结束”
- 松开：结束录音，音频上传并进入处理状态
- 处理中：按钮禁用，显示“AI 思考中…”

#### 4.4.6 状态提示

在聊天区域顶部或底部显示当前状态：
- 聆听中…
- 识别语音中…
- 理解需求中…
- 绘制图片中…
- 你可以继续描述来修改画面

#### 4.4.7 图片生成后的操作

当 AI 生成并展示图片后，界面进入“结果确认态”，用户有三种选择：

**选择 1：继续修改当前图片**
- 用户直接按住底部【按住说话】按钮，语音说出修改要求，例如：
  - “把猫改成黑色的。”
  - “背景换成星空。”
  - “更写实一点。”
  - “把猫向上放一点。”
- 系统保持当前 `session_id` 和上下文，进入修改模式。
- 后端调用 **通义万相图像编辑 API**（`wanx2.1-imageedit`），以当前图片为底图进行编辑。
- AI 回复文字确认，并返回更新后的图片。

**选择 2：整张重画**
- 界面在图片下方显示操作卡片：
  - 【整张重画】
- 用户点击【整张重画】按钮后：
  - 后端清空当前图片参数，但保留会话上下文。
  - 用户语音输入新的完整描述。
  - 后端调用文生图 API 重新生成一张新图片。
- 此操作适用于用户对当前图片整体不满意，希望从头开始的情况。

**选择 3：结束当前对话，开启新对话**
- 界面在图片下方或底部输入区上方显示操作卡片：
  - 【开启新对话】
- 用户点击【开启新对话】按钮后：
  - 弹出二次确认：“确定要结束当前对话吗？当前图片和修改记录将清空。”
  - 确认后，后端清除当前会话上下文，前端清空聊天记录。
  - 界面回到初始状态，等待用户新的语音描述。

**语音识别兜底**：
- 如果用户在图片生成后说“好了”、“结束了”、“新开一个”等，
  前端识别为“开启新对话”意图，弹出确认后清空上下文。
- 用户语音说“重新画一张”等时，可提示用户点击【整张重画】按钮，避免语音误触发。

#### 4.4.8 前端会话状态

前端维护以下状态以控制界面：

```ts
type AppState =
  | 'idle'           // 初始状态，等待用户开始
  | 'recording'      // 录音中
  | 'processing'     // 上传/识别/理解/生成中
  | 'clarifying'     // 需要用户补充信息
  | 'image_ready'    // 图片已生成，等待用户选择下一步
  | 'modifying';     // 用户正在修改当前图片
```

状态切换规则：
- `idle` → 按住录音 → `recording`
- `recording` → 松手 → `processing`
- `processing` + 信息不足 → `clarifying`
- `processing` + 生成完成 → `image_ready`
- `image_ready` + 用户语音修改 → `modifying` → `processing` → `image_ready`
- `image_ready` + 点击“开启新对话” → `idle`

**语音触发新对话**：
- 当处于 `image_ready` 状态时，如果用户说“好了”、“结束了”、“重新画一张”、“新开一个”等，
  前端可直接识别为新对话意图，弹出确认后清空上下文。
- 也可以在 Stage 1 LLM 中增加意图识别字段，由 LLM 判断用户是想修改还是想结束对话。

### 4.5 修改请求的执行策略

当用户针对已生成图片提出修改意见时，系统的处理原则是：**基于原图进行图像编辑，而不是完全重新生成**。

#### 4.5.1 核心方案：通义万相指令编辑

通义万相提供 `wanx2.1-imageedit` 模型的 `description_edit` 功能，支持以原图为底图，通过文字指令修改画面内容。

调用参数：
- `model`: `wanx2.1-imageedit`
- `function`: `description_edit`
- `base_image_url`: 当前图片 URL（要修改的原图）
- `prompt`: 简短明确的修改指令（由 `editPromptBuilder` 直接基于用户输入生成，避免 Stage 2 扩写偏离原意）
- `negative_prompt`: 可选，包含要去掉的元素和通用质量负面词

示例：
```json
{
  "model": "wanx2.1-imageedit",
  "input": {
    "function": "description_edit",
    "base_image_url": "https://.../current-cat.png",
    "prompt": "一只黑色的猫穿着宇航服，站在月球表面插旗，背景是星空和地球，卡通插画风格"
  }
}
```

#### 4.5.2 常见修改类型与执行方式

| 用户修改意图 | Stage 1 处理 | 调用方式 |
|--------------|--------------|----------|
| “把猫改成黑色的。” | 更新 `subject` / `color_tone` | 图像编辑 API，以原图为底图 |
| “背景换成星空。” | 更新 `scene` | 图像编辑 API，以原图为底图 |
| “更写实一点。” | 更新 `style` | 图像编辑 API，以原图为底图 |
| “加一只狗。” | 在 `subject` / `details` 中增加狗 | 图像编辑 API，以原图为底图 |
| “去掉那棵树。” | 从 `details` 中移除树 | 图像编辑 API，以原图为底图 |
| “把猫向上放一点。” | 更新 `composition` | 图像编辑 API，Prompt 中强调位置 |
| “整张重画。” | 清空当前参数，按新描述重新开始 | 文生图 API，从头生成 |

#### 4.5.3 位置/构图类修改的处理

对于“把猫向上放一点”这类位置修改：

1. Stage 1 LLM 识别到用户在调整构图，更新 `composition` 字段。
2. Stage 2 LLM 生成完整画面描述，并在其中加入位置描述，例如：
   - “猫位于画面的上半部分”
   - “猫的位置偏上，留出下方空间”
   - “upper composition, cat positioned higher in the frame”
3. 调用图像编辑 API，以原图为底图进行编辑。

**效果预期**：
- 图像编辑 API 会尽量保留原图的整体构图、风格和主要元素。
- 位置调整是“语义级”的，模型会尝试把猫往上放，但无法保证像素级精确。
- 如果位置调整效果不理想，用户可以进一步说“再往上一点”继续优化。

#### 4.5.4 重新生成 vs 编辑的选择

系统默认使用图像编辑 API 处理修改，仅通过按钮触发重新生成：

| 用户操作 | `edit_mode` | 系统行为 |
|----------|-------------|----------|
| 语音说“把猫改成黑色”、“背景换星空”、“加一只狗”、“把猫向上移”等 | `image_edit` | 使用图像编辑 API，以原图为底图修改 |
| 点击【整张重画】按钮 | `regenerate` | 使用文生图 API，从头生成 |

判断逻辑：
- 默认情况下，用户在已有图片后的语音输入都视为修改意图，`edit_mode: "image_edit"`。
- 只有当用户点击【整张重画】按钮时，`edit_mode: "regenerate"`。
- 前端在调用 `/api/generate` 时，根据用户是否点击按钮传入 `edit_mode` 参数。
- Stage 1 LLM 的 `edit_mode` 字段用于接收并遵循前端传入的模式。

### 4.6 错误处理策略

本项目采用极简错误处理策略：

| 错误类型 | 处理方式 |
|----------|----------|
| ASR 识别失败 / 识别为空 | 前端提示“没有听清，请重新输入”，用户可再次录音 |
| LLM 解析失败 | 不设置专门兜底，请求失败时前端显示通用错误提示 |
| 图像生成失败 | 不设置专门兜底，前端显示通用错误提示 |
| 网络请求失败 | 前端显示通用错误提示 |

说明：
- 仅 ASR 失败需要用户重新输入，因为语音输入是核心交互入口。
- 其他环节的失败以简单提示为主，不阻塞用户继续操作。
- 如需增强健壮性，可在后续迭代中补充重试、降级等机制。

## 判断规则

### 1. 必须明确的信息（缺失时必须追问）
- subject（主体）：用户想画什么？必须有清晰主体。
- style（风格）：如果用户完全没提风格，必须追问。

### 2. 强制生成语义
如果用户表达“不想继续补充”、“用当前信息直接画”、“随便/就这样吧”等意图，请设置 `force_generate: true` 并返回 `status: "complete"`。

常见的强制生成表达：
- “随便吧。” / “随便，就这样吧。”
- “直接画。” / “就按这个画。”
- “不补充了。” / “可以了。”
- “随便你。” / “你决定。”

当 `force_generate: true` 时：
- 即使 `style` 或 `scene` 缺失，也**不再追问**。
- 对缺失字段使用合理默认值或根据已有信息推断。
- 在 `response` 中告知用户“好的，我用当前信息直接为你生成。”

### 3. 可以智能推断的信息（缺失时不一定追问）
- scene（场景）：如果未提及，可以推断为“简洁背景”或“适合主体的常见场景”，不强制追问。
- color_tone（色调）：如果未提及，可以根据 mood 和 style 推断，不强制追问。
- composition（构图）：可以推断为“中景构图，主体在画面中央”。

### 4. 需要追问的模糊情况
- 用户说“画一只猫” → 追问风格和场景。
- 用户说“画风景” → 追问具体是什么风景（山/海/城市）和风格。
- 用户说“好看一点” → 追问“好看”具体指什么（更鲜艳？更写实？更简洁？）。
- 用户说“改一下” → 追问具体改哪里。

### 5. 用户意图是“修改当前图片”时的处理
如果上下文中有当前图片的参数，且用户说“把猫改成黑色”、“背景换成星空”、“更写实一点”等，
请基于已有参数做局部更新，返回 status=complete，并在 extracted 中体现修改后的完整参数。

`edit_mode` 的处理：
- 语音输入的修改请求，默认视为基于原图的编辑，`edit_mode: "image_edit"`。
- “整张重画”由前端按钮触发，`edit_mode: "regenerate"` 由前端传入，LLM 遵循即可。
- 如果用户语音说“重新画一张”等模糊表达，可设置为 `edit_mode: "image_edit"`，由前端引导用户点击【整张重画】按钮。

修改规则：
- “把 X 改成 Y” → 更新对应的字段，`edit_mode: "image_edit"`。
- “更 XX 一点” → 微调 style / mood / color_tone 等，`edit_mode: "image_edit"`。
- “加 XX” / “去掉 XX” → 更新 `subject` 或 `details`，`edit_mode: "image_edit"`。
- “把 XX 向上/下/左/右移动” → 更新 `composition`，`edit_mode: "image_edit"`。
- “换个风格” → 只更新 style，保留 subject 和 scene，`edit_mode: "image_edit"`。

### 5.2 Stage 2 Prompt 扩写策略

> 用于将 Stage 1 提取的结构化参数，转换为通义万相的高质量画面描述 Prompt。

## 6. API 接口设计（草案）

### 6.1 上传语音并处理（生成或编辑）

```http
POST /api/generate
Content-Type: multipart/form-data
```

请求参数：
| 字段 | 类型 | 说明 |
|------|------|------|
| audio | File | 录音文件 |
| session_id | string | 会话 ID，用于上下文 |
| context | JSON string | 当前上下文 |
| edit_mode | string | 可选，默认 `"image_edit"`。前端点击【整张重画】按钮时传入 `"regenerate"` |

后端处理流程：
1. ASR 转文字。
2. Stage 1 LLM 提取参数，并根据前端传入的 `edit_mode` 决定后续调用：
   - 首次生成或 `edit_mode: "regenerate"` → 调用文生图 API
   - `edit_mode: "image_edit"` → 调用图像编辑 API，传入当前图片作为底图
3. 返回图片 URL 或澄清问题。

响应：
```json
{
  "status": "complete" | "need_clarification" | "error",
  "force_generate": false,
  "edit_mode": "image_edit" | "regenerate",
  "extracted": { /* 结构化参数 */ },
  "clarification_question": "",
  "suggestions": [],
  "response": "文字反馈文本",
  "image_url": "",
  "updated_context": { /* 更新后的上下文 */ }
}
```

### 6.2 文本直接生成（调试用）

POST /api/generate-text
Content-Type: application/json

{
  "text": "画一只在月球上插旗的卡通橘猫",
